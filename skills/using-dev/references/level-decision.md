# Level Decision — L0/L1/L2/L3 判定细则

主 SKILL.md 给了 L0-L3 的概要。本文件提供：判定流程、信号细则、关键词映射、PM 类需求的特殊处理、以及用户主动降级的处理。

## 判定流程（伪代码）

```
1. 读用户原话 + git status，推断改动范围
2. 显式信号优先：架构 / 数据模型 / 公共 API → L3
                 新功能 / 跨模块 → L2
                 单 bug / 单文件改动 → L1
                 typo / 日志 / 注释 / 格式 → L0
3. 模糊时取较低档；在开场确认里给出"要升到 L[N+1] 吗？"的选项
4. 用户主动降级 / 升级，按用户优先
5. 永远不"自作主张升级"——升级必须经过用户确认
```

## L0-L3 信号细则表

| 等级 | 触发信号 | 边界 case → 升档 |
|------|---------|------------------|
| **L0** | 单行改动 / typo / 改注释 / 调日志级别 / 仅 format | 涉及多个文件的批量格式化 → L1；改动改变行为（即使一行）→ L1 |
| **L1** | 单 bug 修复 / 单功能小调整 / 改一个函数 / 加一个工具方法 | 修复牵动多个调用方 → L2；引入新依赖 → L2 |
| **L2** | 新功能 / 跨模块改动 / 重构非核心模块 / 新增第三方依赖 | 涉及对外接口 / 公共 SDK → L3；牵动数据迁移 → L3 |
| **L3** | 公共 API 变更 / 数据模型变更 / 架构调整 / 安全相关 / 模块拆分 / 新建子系统 | — |

## 常见关键词 → 等级映射

| 用户说 | 默认等级 | 升档信号 |
|--------|----------|----------|
| "改个 typo"、"调日志"、"加个注释" | L0 | 多文件批量改 → L1 |
| "格式化代码"、"整理一下" | L0 | 涉及大量文件且影响 diff review → L1 |
| "修个 bug"、"这里报错了" | L1 | 跨模块 / 多调用方 → L2 |
| "加个工具函数"、"补个小方法" | L1 | 暴露给外部 → L2 |
| "加个功能"、"实现 X"、"做一个 Y" | L2 | 改公共 API → L3 |
| "重构 X 模块" | L2 | 核心模块 / 跨服务 → L3 |
| "调整架构"、"改数据模型"、"拆 X 服务" | L3 | — |
| "改一下安全策略"、"加权限校验" | L3 | — |
| "性能优化" | L1（局部）/ L2（系统级） | 涉及架构 → L3 |

## PM 类需求特殊处理

PM 类 issue 来自 `github-product-manager`（template 与工程类不同），识别信号：

- issue body 含 `MVP 定义` 字段
- 标签里有 `feature` / `pm-request` / `needs-design`

**处理规则：**

1. **默认 L2 起步**——PM 类需求范围通常较大，但具体实现由 MVP 决定
2. **必读 issue body 的 `MVP 定义` 字段**——这是判定"做对没 / 做多了没"的唯一依据
3. **切片实现**：
   - MVP 内的能力 → 当前 L2 流程一次做完
   - 超出 MVP 的能力 → **必须**开 follow-up issue 而不是塞进当前 PR
4. 若 issue 标 `needs-design` → 升 L3，先走 `architecture-designer`

## 降级请求处理

用户随时可以主动降级，按下表响应：

| 用户说 | 跳过 | 仍保留 |
|--------|------|--------|
| "别建 issue 直接改" | `github-create-issue` / `github-issue-workflow` | `friendly-*` + 本地 commit 规范 |
| "不需要 plan，直接写" | `writing-plans` | `friendly-*` + issue 流程（如已建）+ 复盘 |
| "不复盘" | `self-improving` | — |
| "直接改" / "快点" | 所有 superpowers + issue 流程 | `friendly-*` 规范（永远保留） |
| "降到 L[N-1]" | 按对应等级的链路 | — |

**重要：** 用户拒绝建 issue（L2/L3）时，**必须**告知风险一句话：
> "好，跳过 issue。注意：PR 不会自动 close issue，追溯断链，复盘时缺乏锚点。"

然后继续执行——**不要**反复劝说。

## 升档判定的兜底

当判定难下、信号矛盾时：

1. **先取较低档**——少做一步比多做一步危害小
2. 在开场确认里**显式给出升档选项**："判定为 L1，要升到 L2 吗？"
3. 让用户拍板——不要在内部纠结超过 1 轮

## 自检清单

每次开场判定后，用这张清单自检：

- [ ] 是否检测了语言？
- [ ] 是否检测了 GitHub remote？
- [ ] 是否给出明确的等级 L[N]？
- [ ] 是否给用户至少一个降级 / 升级 / 跳过 issue 的 opt-out 选项？
- [ ] 用户没明确同意之前，是否克制住了"直接动手"的冲动？
