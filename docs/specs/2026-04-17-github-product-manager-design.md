# github-product-manager Skill 设计文档

## 概述

一个"产品经理"角色的 skill，通过多轮对话帮助用户将模糊的想法、一句话需求或部分思考过的功能构想，深度分析为结构化的产品需求 issue，并提交到 GitHub。

面向所有角色（开发者、非技术人员、产品经理自身），适应从一句话到多段描述的各种输入复杂度。

## 文件结构

```
skills/github-product-manager/
  SKILL.md                        # 主流程文档
  references/
    issue-template.md             # 产品需求 issue 模板
    question-framework.md         # 追问维度与问题库
```

## 触发条件

description: 分析用户需求，追问补充想法，产出结构化的产品需求 GitHub issue。触发词："我想做"、"加个功能"、"用户需要"、"需求分析"、"产品需求"、"feature request"、"product requirement"、"analyze requirement"

## 与现有 Skills 的关系

- **独立于 `github-create-issue`** —— 拥有自己的产品需求模板，不复用 bug/架构模板
- **不依赖 `github-issue-workflow`** —— 该 skill 负责"处理已有 issue"，方向相反
- **可被链式调用** —— 未来其他 skill 发现需要提产品 issue 时可引导至本 skill

## 核心流程

### 阶段 1：项目上下文探索

Agent 自动执行，无需用户参与：

1. 扫描项目结构（文件树、README、关键配置文件）
2. 查看最近 10-20 条 git commits，了解项目动态
3. `gh issue list --state open` 查看已有 open issues
4. `gh pr list --state all --limit 10` 查看近期 PR

产出一段简短的项目上下文摘要（3-5 句话），展示给用户确认理解是否正确。

### 阶段 2：需求追问与深度分析

读取 `references/question-framework.md` 获取追问维度，然后：

- 一次一个问题，多选项优先
- 按维度逐步深入（详见追问框架）
- 根据用户回答的复杂度自适应跳过不必要的维度
- 追问过程中如果发现需求包含多个独立子需求，主动提出拆分建议

### 阶段 3：需求整理与 Issue 草稿

追问结束后，读取 `references/issue-template.md`，将分析结果填入模板，生成完整 issue 草稿。同时给出：

- 建议的标签（从预定义标签体系中选择）
- 建议的优先级（P0-P3）
- 如果需要拆分，列出所有子 issue 的标题和简述

### 阶段 4：用户预览与修改

向用户展示完整的 issue 预览，包括标题、标签、优先级、正文。用户可以：

- 确认直接提交
- 要求修改某个部分（agent 按修改意见调整后重新预览）
- 要求拆分或合并

### 阶段 5：提交

用户确认后，通过 `gh issue create` 提交。如有多个 issue，逐一创建。创建完成后返回所有 issue 的 URL。

## 产品需求 Issue 模板

区别于 `github-create-issue` 面向 bug/架构的模板，采用产品视角：

```markdown
## 用户故事
作为 [角色]，我希望 [做什么]，以便 [获得什么价值]。

## 背景与动机
[这个需求从哪来？解决什么问题？不做会怎样？]

## 用户场景
[具体的使用场景描述，1-3 个典型场景]
1. 场景一：...
2. 场景二：...

## 当前状态
[项目中现有的相关功能/代码是什么样的？有什么差距？]

## 预期行为
[详细描述期望的功能表现]
1. ...
2. ...

## 边界与约束
[边界情况、技术约束、兼容性要求、性能要求等]

## 竞品/参考
[有没有参考的实现？竞品是怎么做的？]

## 影响范围
[哪些模块/功能会受影响？]

## MVP 定义
[最小可行版本包含什么？什么可以后续迭代？]

## 验收标准
- [ ] 标准 1
- [ ] 标准 2

## 优先级建议
[P0-P3 + 理由]
```

### 模板裁剪规则

| 需求类型 | 可省略 |
|---------|--------|
| 简单功能 | 竞品/参考、边界与约束 |
| 改进优化 | 用户故事（如果是内部改进）|
| 大型功能 | 无，全部必填 |
| 体验改善 | MVP 定义（如果不需要分期）|

### 标签体系

产品需求使用独立标签，与 `github-create-issue` 的标签互补：

| 标签 | 颜色 | 说明 |
|------|------|------|
| `feature` | `1d76db` | 新功能需求 |
| `enhancement` | `0e8a16` | 已有功能改进 |
| `ux` | `f9d0c4` | 用户体验改善 |
| `needs-design` | `d4c5f9` | 需要进一步设计 |
| `P0-critical` | `b60205` | 最高优先级 |
| `P1-important` | `d93f0b` | 高优先级 |
| `P2-normal` | `fbca04` | 正常优先级 |
| `P3-nice-to-have` | `c5def5` | 低优先级 |

## 追问框架

### 追问维度（按顺序推进）

| 阶段 | 维度 | 核心问题 | 何时跳过 |
|------|------|---------|---------|
| 1 | 核心意图 | 你想解决什么问题？/ 你想实现什么？ | 永不跳过 |
| 2 | 目标用户 | 谁会用这个功能？在什么情况下用？ | 用户已明确说明时 |
| 3 | 用户场景 | 描述一个典型的使用流程 | 永不跳过 |
| 4 | 痛点/现状 | 现在是怎么做的？哪里不好？ | 全新功能且无替代方案时 |
| 5 | 期望行为 | 理想情况下它应该怎么工作？ | 永不跳过 |
| 6 | 边界情况 | 异常情况怎么处理？有什么限制？ | 简单需求 |
| 7 | 与现有功能关系 | 会影响或依赖哪些已有功能？ | 完全独立的新功能 |
| 8 | 竞品/参考 | 有没有见过类似的好的实现？ | 用户明确无参考时 |
| 9 | 优先级与范围 | 这件事多紧急？MVP 包含什么？ | 永不跳过 |

### 追问策略规则

1. **多选项优先** —— 每个问题尽量提供 A/B/C/D 选项，降低用户认知负担
2. **利用项目上下文** —— 结合阶段 1 扫描到的代码和 issues，提出具体的选项而非抽象提问。例如不问"会影响哪些模块"，而是"我看到项目有 X、Y、Z 模块，这个需求主要影响哪个？"
3. **渐进深入** —— 前 3 个维度快速建立基本画面，后续维度逐步补充细节
4. **自适应退出** —— 当收集到的信息已经足够填充 issue 模板的必填项时，主动提出"我觉得信息够了，要进入整理阶段吗？"，而不是机械地问完所有维度
5. **拆分检测** —— 追问过程中如果发现用户描述的其实是 2+ 个独立需求，在当前问题结束后立即提出拆分建议，让用户选择先聚焦哪一个

### 追问风格

- 像一个有经验的产品经理同事跟你聊天，而不是填表
- 对用户的回答做简短的复述/确认，再接下一个问题
- 如果用户的回答模糊，用具体化的追问帮助用户想清楚，而非直接接受模糊答案

## Issue 提交机制

### Title 规范

格式：`[模块] 简述需求`

示例：
- `[auth] 支持第三方 OAuth 登录`
- `[editor] 添加 Markdown 实时预览`
- `[skills] 产品经理式需求分析流程`

### 提交命令

```bash
gh issue create \
  --title "[模块] 简述" \
  --label "label1,label2" \
  --body "$(cat <<'EOF'
## 用户故事
...

## 背景与动机
...

（其余模板 section）

## 验收标准
- [ ] ...

## 优先级建议
...
EOF
)"
```

### 标签前置检查

提交前检查标签是否存在：`gh label list | grep <label>`
不存在则创建：`gh label create <label> --color <color> --description "<desc>"`

### 多 Issue 拆分流程

1. 追问阶段识别出需要拆分
2. 列出所有子需求的标题和简述，用户确认
3. 逐一创建，每个 issue 的"关联"section 互相引用
4. 全部创建后汇总返回所有 URL
